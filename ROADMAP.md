# 이음(화성복지플랫폼) 발전 전략 로드맵

> **비전** — 흩어진 복지 데이터를 모아 **필요한 사람과 제공할 수 있는 사람을 잇는다.**
> 이음은 수요자(취약계층) · 서비스 제공 업체 · 후원 업체를 매칭하는 **3자 플랫폼**이 된다.
> 이 모든 것의 심장은 **데이터베이스**다.

---

## 큰 그림 — 3면(three-sided) 매칭 마켓플레이스

```
   [수요자]  "이런 서비스 받고 싶어요"   →  신청(needs)
   취약계층      ↑                              ↓  지역-지역 매칭
 (다문화·노인·   |            ┌───────────┐
  아동청소년·    └────────────│   이음     │────────────┐
  한부모)                     │  (플랫폼)  │            │
                              │  = DB·매칭 │            │
   [제공업체]  "내가 제공할게요" ←─ 매칭 ──┘   실적관리   │  후원 연계
   복지·돌봄·                                            ↓
   교육·일자리                                      [후원 업체]
   업체·기관                                    "여기에 후원할게요"
```

핵심 원리: **수요(needs)와 공급(offers)을 데이터로 매칭**해 서로의 필요를 채운다.
이음은 직접 서비스를 제공하지 않고, **연결·검증·실적관리·후원중개**를 하는 신뢰 계층이다.

---

## 단계별 전략

### 0단계 · 데이터 시각화 (현재 — 대부분 완료 ✅)
- 전국/화성 복지 데이터 지도·대시보드, 4개 구·29개 읍면동 계층, 서비스 디렉터리.
- **목적**: 신뢰 확보 + 지역 현황 파악 + 홍보 자산. (지금까지 만든 것)

### 1단계 · 기본 데이터 정리 (데이터 표준화)
- **목표**: 흩어진 공공·통계 데이터를 하나의 표준 스키마로 정리.
- 작업: KOSIS·행안부·공공데이터포털 지표를 지역(시도→시군구→읍면동) 기준으로 정규화(이미 착수). 복지 서비스 분류 체계(카테고리·태그) 확정.
- **산출물**: 지역 마스터 데이터 + 서비스 분류 표준. (코드로는 `hwaseong-regions.js` 같은 정규화 모듈)

### 2단계 · 지역 실데이터 정리 및 매칭 준비
- **목표**: "추정치"를 **실제 지역 데이터**로 교체하고, 수요-공급 매칭의 뼈대를 만든다.
- 작업:
  - data.go.kr/지자체 협약으로 읍면동 단위 복지대상·자원 실값 확보(수급·장애·독거 등).
  - **서비스 제공 자원 대장** 구축: 지역 복지기관·업체·자원봉사·후원처 목록(연락처·제공가능 서비스·지역·정원).
  - 수요-공급을 잇는 **매칭 규칙 정의**(지역·카테고리·긴급도·정원).
- **산출물**: 실데이터 DB + 제공자원 대장 + 매칭 규칙 v1.

### 3단계 · 복지 지원 서비스 확대 + 이음 앱 (전환점 🔑)
- **목표**: 정적 사이트 → **로그인·신청·매칭이 되는 서비스**로. 이음 앱/웹앱 출시.
- 작업:
  - **백엔드 + DB 도입**(아래 아키텍처 참조). 사용자 인증·데이터 저장.
  - **카카오톡 로그인(카카오싱크)** 으로 가입 마찰 최소화 + **카카오톡 채널** 개설(공지·알림·상담).
  - 취약계층(다문화·노인·아동청소년·한부모) **연락망 구축** — 동의 기반 개인정보 수집(⚠ 아래 법·개인정보 필수).
  - 수요자: **"이런 서비스 받고 싶어요" 신청란** → 원하는 복지 서비스 등록.
  - PWA(이미 구축)로 "홈 화면 추가" 앱 경험 제공, 이후 필요 시 네이티브 앱.
- **산출물**: 로그인·신청 가능한 이음 앱(웹앱) + 카톡 채널 + 취약계층 연락망 DB.

### 4단계 · 매칭 마켓플레이스 가동
- **목표**: 신청(수요) ↔ 제공(공급)을 **지역 대 지역으로 매칭**.
- 작업:
  - 제공업체 온보딩: 가입·자격검증·**자체 홍보 페이지** 제공(이음 룰·법 테두리 내).
  - 매칭 플로우: 수요자 신청 → 이음이 후보 업체에 알림 → 업체 **"내가 제공할게요"** → 수요자 확인 → 연계 완료.
  - **실적 관리**: 제공 건수·만족도·이행률 대시보드(업체 신뢰도 등급).
  - 안전장치: 신원확인, 이용약관, 분쟁·환불·품질 룰, 법적 책임 경계 명시.
- **산출물**: 양방향 매칭 엔진 + 업체 홍보/실적 관리 + 신뢰 등급.

### 5단계 · 후원 생태계 + 지속가능성
- **목표**: 후원 업체를 연결해 **후원→제공→수요**의 선순환 완성.
- 작업:
  - 후원 업체 온보딩: 이음 통해 **"원하는 곳에 후원"**(지역·분야·대상 지정 후원).
  - 후원-제공-수요 3자 매칭, 후원 사용처·성과 투명 리포트(기부금 영수증·임팩트 대시보드).
  - 이음 수익모델(수수료·후원 운영비·정부 위탁·데이터 리포트)로 자립.
- **산출물**: 후원 매칭 + 임팩트 리포트 + 지속가능 운영 모델.

## 🗄️ 데이터베이스 아키텍처 (플랫폼의 심장)

핵심 엔티티(테이블)와 관계:

| 엔티티 | 핵심 필드 | 관계 |
|---|---|---|
| **User(사용자)** | id, 카카오ID, 역할(수요자/제공자/후원자/운영자), 지역(읍면동), 동의내역 | 1:N Request/Offer |
| **Beneficiary(수요자 프로필)** | 취약유형(다문화·노인·아동·한부모…), 가구정보(민감) | User와 1:1 |
| **ServiceRequest(신청)** | 원하는 서비스 카테고리, 지역, 긴급도, 상태(대기/매칭/완료) | User→Provider |
| **Provider(제공업체)** | 기관명, 자격/인증, 제공 가능 서비스·지역·정원, 신뢰등급 | 1:N Offer |
| **Offer(제공)** | 어떤 Request에 응답, 조건, 상태 | Request↔Provider |
| **Match(매칭)** | request_id, provider_id, 확정일, 이행상태, 평가 | 실적의 원천 |
| **Sponsor(후원업체)** | 후원처, 후원 조건(지역·분야·대상) | 1:N Sponsorship |
| **Sponsorship(후원)** | 대상 Provider/Match, 금액·현물, 영수증, 성과 | 후원 연계 |
| **AuditLog(감사)** | 개인정보 접근·처리 이력 | 법적 대응 |

- **매칭 쿼리**: `지역 일치 + 카테고리 일치 + (정원 여유) + 신뢰등급` 으로 후보 정렬.
- **원칙**: 개인 식별정보(민감정보)와 매칭용 정보를 **분리 저장**, 최소권한 접근, 암호화.

## ⚖️ 개인정보·법규 (취약계층 데이터라 특히 중요)

- **법적 근거**: 개인정보보호법(민감정보·고유식별정보 별도 동의), 사회복지사업법, 사회적협동조합(협동조합기본법) 지위 활용.
- **필수 절차**: ①수집·이용 목적 명시 ②정보주체(또는 법정대리인) **별도 동의** ③최소 수집 ④보관기간·파기 정책 ⑤암호화·접근통제·감사로그 ⑥위탁 시 계약·재위탁 통제.
- **취약계층 특례**: 아동·고령·장애인은 대리 동의·쉬운 설명(easy-read) 필요. 다문화는 다국어 동의.
- **카카오 로그인 데이터**: 카카오싱크 최소 항목만, 목적 외 사용 금지. 알림톡은 정보성/광고성 구분.
- **권장**: 초기부터 개인정보 영향평가(PIA) 관점 반영, 자문(변호사·개인정보 전문가) 확보. 지자체 협약으로 공공 데이터 합법 연계.

## 🛠️ 기술 전환 경로 (현재 → 목표)

| 단계 | 지금 | 필요 전환 |
|---|---|---|
| 0~2단계 | 정적 사이트(GitHub Pages) + 조회용 프록시 | 정적 유지, 데이터 모듈 정규화 |
| 3단계 | — | **백엔드 + DB + 인증**(예: Supabase/Firebase 또는 Node+PostgreSQL), 카카오 로그인, PWA 앱화 |
| 4~5단계 | — | 매칭 엔진(서버 로직·알림), 관리자 콘솔, 결제/후원, 감사·보안 강화 |

- 현재 자산 재활용: PWA(설치형)·프록시(API 연동)·지역 데이터 모듈·서비스 디렉터리.

## 📣 이음 앱 활성화(홍보·가입 유도)

- **신뢰 자산 먼저**: 지금의 데이터 시각화 사이트를 "이음이 지역을 안다"는 증거로 활용.
- **오프라인 접점**: 경로당·행정복지센터·복지관·다함께돌봄센터와 제휴해 가입 안내(고령·다문화는 대면이 핵심).
- **카카오톡 채널**: 채널 추가 → 공지·상담·신청 안내. QR·현수막·통장/이장 네트워크.
- **초기 KPI**: 채널 친구 수, 신청 건수, 제공업체 등록 수, **매칭 성사율**, 재이용률.

> 이 로드맵은 이음의 최종 목표(3자 매칭 플랫폼)로 수렴한다. 각 단계는 다음 단계의 데이터·신뢰를 쌓는다.
