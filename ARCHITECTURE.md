# 이음 플랫폼 — 기술 기반 전략 (보안 · 데이터베이스 · 트래픽)

> **전제** — 이 사업은 화성시 → 경기도 → 전국으로 확산된다.
> **원칙** — "처음 설정이 전부다." 확산을 못 하게 만드는 결정을 **초기에 하지 않는 것**이 핵심.
> 목표: **확산 대비 견고한 기초(scale-ready foundations)를 지금 깔되, 규모는 단계적으로 키운다.**

---

## 0. 관통 원칙 (모든 결정의 기준)

1. **지역(Region)을 1급 시민으로.** 모든 데이터에 행정표준코드(시도·시군구·읍면동)를 붙인다. → 화성만 켜기 → 경기 → 전국이 "코드 추가"로 끝난다(구조 변경 아님).
2. **개인정보는 설계로 보호(Privacy by Design).** 민감정보 분리·최소수집·동의·암호화·감사는 나중에 못 붙인다. **Day 1에.**
3. **무상태(stateless)·수평확장.** 서버는 언제든 늘릴 수 있게. 상태는 DB·캐시·큐에.
4. **관리형 우선(Managed first).** 소규모 협동조합이 인프라 운영에 매몰되지 않게, 처음엔 관리형 서비스로 빠르게. 규모가 커지면 이관.
5. **데이터 국내 보관.** 개인정보는 국내 리전에(국외이전 규제 회피).
6. **인프라 코드화(IaC)·환경 분리(dev/staging/prod).** 재현 가능·안전한 배포.

## 1. 데이터베이스 전략

### 선택: PostgreSQL (관계형)
- 매칭·거래·관계·정합성에 강함. **Row-Level Security(RLS)** 로 지역·역할별 접근통제를 DB단에서 강제 → 다지역·개인정보 플랫폼에 최적.
- 초기 관리형: **Supabase**(Postgres + 인증 + 스토리지 + RLS + 실시간) 또는 AWS RDS / Cloud SQL. 소규모엔 Supabase가 빠르고 저렴.

### 확산을 여는 3가지 설계 (지금 확정)
- **지역 파티셔닝**: 모든 핵심 테이블에 `sido_code, sigungu_code, emd_code`. 지역 스코프 질의·단계적 오픈·후일 샤딩의 기반.
- **민감정보 분리**: PII(취약유형·가구정보·연락처)는 **별도 스키마/테이블 + 암호화 + 최소권한**. 매칭용 데이터는 가명 ID로만 참조.
- **RLS 정책**: 수요자는 본인 데이터만, 제공자는 자기 지역의 신청만, 운영자만 전체. 앱·업체가 늘어도 안전.

### 확장 경로 (규모별)
| 규모 | DB 구성 |
|---|---|
| 화성(수천) | 단일 Postgres + 커넥션 풀(PgBouncer) |
| 경기(수만~수십만) | 읽기 복제(Read Replica) + Redis 캐시 |
| 전국(수백만) | 지역 파티셔닝/샤딩(Citus 등) + 분석용 DW 분리 |

- 공통: **자동 백업 + 시점복구(PITR) + 저장 암호화**. 정기 복구 훈련.

## 2. 보안 전략

### 인증·권한
- **카카오 로그인(OAuth/OIDC)** + 자체 세션/JWT. 운영자·제공업체는 **2단계 인증(MFA)**.
- **역할 기반 접근제어(RBAC)**: 수요자 / 제공자 / 후원자 / 운영자 4역할 + RLS로 이중 방어.

### 데이터 보호
- **전 구간 암호화**: HTTPS(TLS) 필수, 저장 암호화(DB·스토리지), 민감필드는 필드 암호화/토큰화. **주민번호 등 고유식별정보는 수집 지양**, 불가피하면 암호화·법적 근거 확보.
- **동의 관리 시스템**: 목적·항목·보관기간별 동의 기록, 철회·파기 자동화. 다국어·easy-read 동의(다문화·고령·아동).
- **감사로그**: 개인정보 접근·처리 전건 기록(AuditLog). 관리자 활동 로깅.

### 애플리케이션·인프라
- **OWASP Top 10 대응**: 입력검증, SQLi/XSS 방지, 인가 검증. **레이트리밋 + WAF + 봇/DDoS 보호(Cloudflare)**.
- **시크릿 관리**: 키는 환경변수·시크릿 매니저에(코드·저장소에 절대 금지 — 현재도 `.env` gitignore로 준수 중). 의존성 취약점 스캐닝.
- **최소권한 IAM·네트워크 분리(VPC)**, 정기 보안점검·모의해킹(규모 커지면).

### 컴플라이언스
- **개인정보보호법·정보통신망법·사회복지사업법** 준수. **개인정보 영향평가(PIA)** 초기 반영.
- 규모 확대 시 **ISMS-P** 인증 로드맵. **침해사고 대응계획(IR)**·비상연락체계 사전 수립.

## 3. 트래픽·확장성 전략

### 계층 분리 (읽기 트래픽 ≠ 쓰기 트래픽)
- **공개 사이트(홍보·조회)**: 정적 + **CDN**(현재 GitHub Pages → 전국 규모 시 Cloudflare/CDN). 정적 콘텐츠는 사실상 무한 확장·저비용. **홍보 폭주 트래픽을 여기서 흡수.**
- **앱 API(로그인·신청·매칭)**: 무상태 서버. 처음엔 **서버리스**(Vercel/Cloud Functions/Supabase Edge)로 자동확장 → 부하 커지면 **컨테이너 오토스케일**(Cloud Run/ECS/K8s)로 이관.

### 부하 흡수 장치
- **캐시**: CDN 엣지 캐시(공개 데이터) + Redis(앱 핫데이터) + DB 질의 캐시.
- **비동기 큐**: 매칭 알림·카톡 발송·리포트·배치는 **큐(SQS/PubSub) + 워커**로 처리 → 사용자 요청을 막지 않음.
- **레이트리밋·오토스케일·우아한 성능저하(graceful degradation)**.

### 운영 가시성 (Observability)
- 지표·로그·트레이스 모니터링 + **에러 추적(Sentry)** + 업타임 감시 + 알림. 용량계획(capacity planning).

### 단계적 오픈으로 리스크 축소
- **지역 플래그**로 화성만 → 경기 → 전국을 점진 오픈. 장애 반경(blast radius)과 비용을 통제하며 확산.

## 5. 단계별 인프라 로드맵 (화성 → 경기 → 전국)

| 단계 | 규모 | 스택 | 보안·운영 |
|---|---|---|---|
| **A. 화성 PoC** | ~수천 | Supabase(Postgres+Auth+RLS+Storage) · Vercel(정적+서버리스) · Cloudflare(CDN/WAF) · 카카오 로그인 | 동의·감사로그·백업·시크릿관리 기본 |
| **B. 경기 확산** | ~수만~수십만 | + 읽기복제 · Redis · 큐/워커 · 모니터링·Sentry | WAF·레이트리밋·정기 보안점검·IR 계획 |
| **C. 전국** | ~수백만 | + 파티셔닝/샤딩(Citus) · 컨테이너 오토스케일 · 멀티AZ · 분석 DW 분리 | ISMS-P 인증 · 전담 보안/SRE · 모의해킹 |

- **현재 자산 재활용**: 정적 사이트(CDN 계층), API 프록시(→ 서버리스로 진화), PWA, 지역 데이터 모듈.
- **비용 원칙**: pay-as-you-grow. 관리형·서버리스로 시작해 트래픽에 비례해 지출.

## 6. 권고 요약

1. **지금**: 데이터 모델에 **지역 코드 + 개인정보 분리 + 역할/RLS**를 넣은 스키마를 확정한다(코드 몇 줄이 아니라 설계 결정).
2. **3단계 착수 시**: Supabase + Vercel + Cloudflare + 카카오 로그인으로 **저비용·확산가능** 기반을 구축한다.
3. **개인정보·보안 설계를 기능과 동시에.** 나중에 붙이면 재설계·법적 리스크.
4. **정적/CDN(공개) ↔ API/DB(앱)** 를 분리해, 홍보 트래픽과 서비스 트래픽을 따로 확장한다.
5. **단계적 지역 오픈**으로 안전하게 화성 → 경기 → 전국.

> 요지: **작게 시작하되, 전국을 감당할 설계 위에서 시작한다.**
> 관계형 DB(Postgres)+RLS, 지역 파티셔닝, 개인정보 분리·동의·감사, 정적/CDN과 API 분리 —
> 이 다섯 가지를 처음에 맞추면 확산은 "지역을 켜는 일"이 된다.
